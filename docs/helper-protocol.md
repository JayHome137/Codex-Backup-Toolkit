# Local Helper Protocol

This document describes the protocol between the Web GUI and the opt-in local helper.

The Web GUI does not execute shell commands by default. `Local Bridge` mode uses this protocol shape with a mock helper transport. `HTTP Helper` mode can call an opt-in local Node helper, but only when the user starts that helper manually.

## Safety Boundary

The helper protocol is allowlist-first. A request can only be created for:

- `doctor`: `./scripts/codexbackup.sh --doctor --target <target>`
- `backup`: `./scripts/codexbackup.sh --target <target>`
- `validate`: `./scripts/codexinstallautomation.sh validate` with `CODEX_BACKUP_LAUNCHD_LABEL=dev.codexbackup.toolkit.test.<target>`

The GUI blocks restore, install, uninstall, status, and any command outside the allowlist. The isolated validate label is intentional: it prevents the GUI from touching a user-installed backup job.

The helper also rejects appended shell commands and common shell command separators. Validate requests must end with exactly `./scripts/codexinstallautomation.sh validate`. Backup requests must end with exactly `./scripts/codexbackup.sh --target <target>`. Environment lines may only contain simple assignments.

Encrypted backup requests are allowed only when the command includes `CODEX_BACKUP_AGE_RECIPIENT` or `CODEX_BACKUP_AGE_RECIPIENT_FILE`.

## HTTP Endpoints

The draft Node helper listens on `127.0.0.1` only. Default port is `37371`.

Start it manually for development validation:

```zsh
node helper/server.mjs
```

Optional port override:

```zsh
CODEX_BACKUP_HELPER_PORT=37372 node helper/server.mjs
```

The GUI currently expects `http://127.0.0.1:37371` when `HTTP Helper` mode is selected.

The GUI `检查助手` action calls only `GET /health`; it does not invoke `/run` and cannot execute shell commands.

### `GET /health`

Returns helper status:

```json
{
  "schema": "codex-backup-helper.v1",
  "version": 1,
  "status": "ok",
  "helper": "node-local-helper",
  "host": "127.0.0.1"
}
```

### `POST /run`

Accepts a helper request JSON body and returns a helper response. The server re-checks the allowlist before running anything, even if the GUI has already checked it.

Unsupported paths and methods return JSON errors. Browser CORS is only opened for local HTTP origins such as `127.0.0.1`, `localhost`, and `::1`.

## Request Schema

```json
{
  "schema": "codex-backup-helper.v1",
  "version": 1,
  "requestId": "cbt_<generated>",
  "createdAt": "2026-06-04T00:00:00.000Z",
  "kind": "backup",
  "command": "CODEX_BACKUP_TARGET=local \\\nCODEX_BACKUP_LOCAL_DIR=\"$HOME/CodexBackups\" \\\n./scripts/codexbackup.sh --target local"
}
```

Fields:

- `schema`: fixed protocol name, currently `codex-backup-helper.v1`.
- `version`: numeric protocol version, currently `1`.
- `requestId`: generated client-side id for correlation and audit logs.
- `createdAt`: ISO timestamp generated by the GUI.
- `kind`: `doctor`, `backup`, or `validate`.
- `command`: the exact command preview that passed the allowlist.

## Response Schema

```json
{
  "schema": "codex-backup-helper.v1",
  "version": 1,
  "requestId": "cbt_<generated>",
  "status": "ok",
  "exitCode": 0,
  "stdout": "Doctor passed.",
  "stderr": "",
  "audit": {
    "commandKind": "doctor",
    "decision": "allowed",
    "helper": "native-helper",
    "startedAt": "2026-06-04T00:00:01.000Z",
    "finishedAt": "2026-06-04T00:00:01.500Z"
  }
}
```

Fields:

- `status`: `ok` or `error`.
- `exitCode`: process-style exit code. For mock helper success this is `0`.
- `stdout`: command output or helper message.
- `stderr`: command error output when available.
- `errorCode`: optional machine-readable error code.
- `audit`: required audit summary for every helper response.

## Error Codes

- `ERR_COMMAND_NOT_ALLOWED`: command failed the GUI allowlist before reaching the helper.
- `ERR_HELPER_UNAVAILABLE`: GUI could not reach or use the helper transport.
- `ERR_HELPER_FAILED`: helper responded but the command failed.

## Current Implementation

The current implementation lives in two places:

- `gui/src/lib/localBridge.ts`: allowlist classification and Local Bridge runner.
- `gui/src/lib/helperProtocol.ts`: request/response types, request builder, mock transport, HTTP transport.
- `helper/server.mjs`: opt-in loopback HTTP helper.
- `helper/allowlist.mjs`: helper-side allowlist.
- `helper/executor.mjs`: shell executor used only after helper-side allowlist approval.

Run helper tests:

```zsh
node --test helper/server.test.mjs
```

The helper server is never started automatically by this repository. The mock transport exists so the UI can show request ids, schema names, command kinds, decisions, helper identity, and exit codes without running shell commands.
